## 分析要求

1. **规则命中解释**：对每个 rule_hits 中的命中项，结合 config_facts 中的 observed_value，用自然语言解释为什么这条规则被触发、具体风险是什么。

2. **语义深度分析**：规则是自动化检测的第一层，LLM 需要做第二层分析：
   - 分析规则未覆盖的潜在风险（如 alias+try_files 路径遍历的具体利用方式、cookie 中 org_code 的业务影响）
   - 对多个配置文件之间的关系做交叉分析（如 ng2 的 8443 server 缺少安全头与 ng1 的 8443 server 有 HSTS 的对比）
   - 判断哪些风险需要联动修复（如 HTTP 不重定向到 HTTPS + cookie 缺 Secure 是一组相关问题）

3. **优先级排序**：结合严重性和实际影响，对风险项重新排序，给出修复优先级。

4. **可执行修复建议**：每个风险项给出具体的 nginx 配置修改示例（完整 directive，不是笼统建议）。
